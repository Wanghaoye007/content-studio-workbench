# PIAS 工作台状态持久化设计

## 1. 目标

修复 `PIAS-P0-001`：工作台显示“已自动保存”，但用户完成审核、移动节点、创建任务或修改结果后刷新页面，状态会回到固定演示数据。

本阶段采用服务端文件持久化，为当前单机 Vite 服务提供可靠的刷新和进程重启恢复，并建立后续迁移数据库时可保持不变的客户端 API 契约。

## 2. 范围

本阶段保存完整 `StudioState`：

- 租户、项目和工作区名称。
- 当前选中场景和工具。
- 用量汇总。
- 素材、场景、连线、任务、结果和审计事件。

以下内容不在本阶段解决：

- 登录、MFA、RBAC 和可信用户身份。
- 租户/项目隔离和跨租户访问控制。
- 多实例共享数据库、备份和灾备。
- Worker 租约、心跳和完整任务状态机。

因此，本阶段完成后只能关闭“刷新和单机重启丢失”这一运行缺陷；系统整体生产验收仍为红色。

## 3. 方案选择

### 3.1 采用方案

通过 Vite Connect 中间件提供工作台状态 API，服务端使用临时文件加原子重命名写入 JSON。前端启动时先读取服务端快照，读取完成后才允许自动保存。

选择该方案的原因：

- 延续现有 Fal Vite 服务端代理和文件持久化模式，改动边界清晰。
- 数据保存在服务端，刷新、关闭页面和单机服务重启后仍可恢复。
- API 与存储实现解耦，后续替换 PostgreSQL 不需要重写前端状态协议。

### 3.2 不采用方案

- `localStorage`：无法提供服务端业务事实、跨设备恢复和并发控制。
- 当前阶段直接引入数据库：会同时牵涉身份、租户、迁移和部署，超过单个缺陷的修复范围。

## 4. 服务端设计

### 4.1 文件位置

默认保存到：

```text
/tmp/pias-image-studio/studio-state.json
```

可通过 `PIAS_STUDIO_STATE_FILE` 覆盖。文件权限为 `0600`，父目录按需创建。

文件格式：

```json
{
  "schemaVersion": 1,
  "revision": 8,
  "updatedAt": "2026-07-21T16:00:00.000Z",
  "state": {}
}
```

`revision` 每次成功写入加一。写入先落到同目录临时文件，再通过 `rename` 原子替换，避免进程中断留下半截 JSON。

### 4.2 API 契约

#### GET `/api/studio/state`

存在已保存快照时：

```json
{
  "schemaVersion": 1,
  "revision": 8,
  "updatedAt": "2026-07-21T16:00:00.000Z",
  "state": {}
}
```

首次运行且文件不存在时返回 `404`：

```json
{
  "error": {
    "code": "STUDIO_STATE_NOT_FOUND",
    "message": "尚未保存工作台状态"
  }
}
```

前端收到该响应后使用 `seedDemoState()`，并将其作为 revision `0` 的首次保存输入。

#### PUT `/api/studio/state`

请求体：

```json
{
  "schemaVersion": 1,
  "expectedRevision": 8,
  "state": {}
}
```

成功返回新的快照元信息：

```json
{
  "schemaVersion": 1,
  "revision": 9,
  "updatedAt": "2026-07-21T16:01:00.000Z"
}
```

若 `expectedRevision` 与服务端当前 revision 不同，返回 `409 STUDIO_STATE_CONFLICT`，不覆盖文件。首次保存要求 `expectedRevision` 为 `0`。

### 4.3 输入约束

- 请求体最大 `5 MiB`。
- `schemaVersion` 必须为 `1`。
- `expectedRevision` 必须为非负整数。
- `state` 必须通过 `StudioState` 运行时结构校验。
- 数组字段、用量数字、必要字符串和任务/审核枚举必须合法。
- 解析错误返回 `400`，体积超限返回 `413`，存储失败返回安全的 `500` 文案。
- 响应和日志不得包含 Fal Key、原始堆栈或完整图片 Data URL。

### 4.4 并发与写入串行化

存储服务内部维护写入 Promise 队列，确保同一进程内的读取 revision、校验和写文件为串行操作。版本不一致时拒绝后到写入，避免两个页面静默覆盖。

该机制只保证当前单进程服务；多实例部署需在数据库阶段改为事务和行版本控制。

## 5. 前端设计

### 5.1 启动流程

`App` 不再同步执行 `seedDemoState()` 作为最终业务状态，而是进入 `loading`：

1. 请求 `GET /api/studio/state`。
2. 返回快照时，使用快照状态和 revision。
3. 返回 `404` 时，使用演示状态和 revision `0`，随后执行首次保存。
4. 网络或服务错误时显示恢复失败页面，提供“重试加载”，不自动创建并覆盖服务端数据。

加载完成前不挂载 `Workbench`，从根源上防止默认演示状态抢先触发自动保存。

### 5.2 自动保存

状态发生变化后等待 `400 ms`，将当前快照和 `expectedRevision` 提交到 PUT 接口。保存期间如果又发生变化，只保留最新待保存快照；当前请求结束后继续保存最新版本。

保存状态分为：

- `loading`：正在恢复工作台。
- `idle`：已加载，尚无待保存变化。
- `saving`：正在保存。
- `saved`：服务端确认成功。
- `error`：网络或服务错误，可重试。
- `conflict`：服务端 revision 已变化，必须重新加载。

顶部状态文案对应显示“正在恢复”“正在保存”“已自动保存”“保存失败”“存在更新冲突”。只有收到服务端成功响应后才能显示“已自动保存”。

### 5.3 冲突处理

收到 `409` 后停止自动重试，保留当前页面内存状态并显示冲突提示。用户可选择“重新加载服务端版本”；本阶段不做自动合并，避免丢失任意一方修改。

## 6. 模块边界

| 模块 | 职责 |
| --- | --- |
| `src/studio/studioStateSchema.ts` | 运行时校验持久化状态，不依赖 HTTP 或文件系统 |
| `src/studio/studioStatePersistence.ts` | 文件读取、原子写入、revision 和串行化 |
| `src/studio/studioStatePlugin.ts` | Vite API 路由、请求限制和安全错误响应 |
| `src/studio/studioStateClient.ts` | 浏览器 GET/PUT 协议和类型化错误 |
| `src/studio/usePersistentStudioState.ts` | 加载、自动保存、冲突和重试状态机 |
| `src/App.tsx` | 根据持久化状态渲染加载、错误或工作台 |
| `src/workbench/Workbench.tsx` | 仅消费保存状态，不直接访问持久化 API |

## 7. 错误处理

| 场景 | 服务端/客户端行为 |
| --- | --- |
| 文件不存在 | 返回 404；前端使用演示状态并首次保存 |
| 文件损坏或 schema 不兼容 | 返回 500/不兼容错误；不覆盖损坏文件 |
| 请求 JSON 非法 | 返回 400 |
| 请求超过 5 MiB | 返回 413 |
| revision 冲突 | 返回 409；前端停止自动保存并提示重新加载 |
| 网络中断 | 当前状态保留在内存，显示保存失败并允许重试 |
| 保存中再次修改 | 当前保存结束后提交最新快照，不丢修改 |

## 8. 测试策略

采用测试驱动实现，至少覆盖：

1. Schema 接受合法 `StudioState`，拒绝缺失字段、非法枚举和非有限数字。
2. 首次 GET 返回 404，首次 PUT 创建 revision `1`。
3. 保存后重新创建存储实例仍能读取相同状态，证明进程重启恢复。
4. 原子写入不会暴露临时或半写文件。
5. 相同 revision 的并发保存只有一个成功，另一个返回冲突。
6. 非法 JSON、超限请求和非法 state 返回对应安全错误。
7. 客户端正确区分未保存、冲突和一般服务错误。
8. `App` 加载完成前不显示工作台，也不保存演示状态。
9. 状态变更后显示“正在保存”，服务端确认后显示“已自动保存”。
10. 浏览器执行审核通过后刷新，状态仍为已通过。
11. 重启 Vite 服务后再次加载，状态仍保持。

## 9. 验收与报告更新

完成本阶段后：

- `PIAS-P0-001` 标记为“单机环境已修复并复验”。
- `DATA-001` 从 `fail/P0` 调整为 `partial/P1`，剩余风险为多实例数据库、备份和灾备。
- 生产总判定仍为红色，因为 `AUTH-001` 与 `ISOLATION-001` 仍是 P0。
- `npm run acceptance:report`、完整测试、生产构建和浏览器复验必须重新执行并归档。

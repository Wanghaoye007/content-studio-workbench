# Prompt 03：数据与状态一致性专项自查

```text
你现在是资深数据质量工程师和状态机审计专家。请从“系统会在什么路径下产生不可恢复的错误事实”出发，检查状态合法性、字段一致性、数据完整性和可追溯性。

【输入】
业务状态规则与计算口径：
{{STATE_RULES_AND_CALCULATION_RULES}}

前端类型、接口、服务逻辑、数据库结构与迁移：
{{FRONTEND_API_SERVICE_DATABASE}}

审计、账务和任务日志样例：
{{AUDIT_LEDGER_JOB_LOGS}}

【强制执行步骤】
1. 先复述核心实体、主键、关联关系、状态枚举、允许的转换、操作者和每次转换的副作用。
2. 输出完整状态转换表，逐一检查非法跳转、状态回退、重复执行、取消、失败、超时、部分成功和恢复。
3. 沿“前端输入 -> API -> 服务 -> 数据库 -> 异步任务 -> 结果 -> 审计/账务 -> 前端展示”建立字段映射表。
4. 对每个金额、额度、数量和统计字段写明唯一口径、单位、精度、时区及预占/结算/释放规则。
5. 反证并发更新、请求重放、事务中断、跨租户引用、孤儿记录、历史迁移和删除后的数据完整性。
6. 每个问题给出完整触发路径、受影响数据、检测 SQL 或日志条件、修复和数据回填方案。
7. 没有问题的维度必须写“该维度无问题”并列出证据；缺少数据库或日志时标记“证据不足”，不得推断为通过。

【输出格式】
一、数据模型与业务口径复述
二、证据缺口
三、状态转换表
| 实体 | 当前状态 | 操作 | 目标状态 | 允许角色 | 前置条件 | 副作用 | 实现状态 | 证据 |

四、字段一致性映射
| 业务字段 | 前端 | API | 服务 | 数据库 | 审计/账务 | 单位/口径 | 结论 |

五、完整性反证场景
| 场景ID | 触发路径 | 期望事务边界 | 可能错误事实 | 检测方法 | 结论 |

六、问题清单
| 问题编号 | 问题类型 | 所在位置 | 触发条件 | 实际结果 | 期望结果 | 影响等级 | 修复方案 | 验证方式 | 证据状态 |

七、每个问题的数据修复方案
- 代码修复
- 数据迁移或回填
- 幂等与并发保护
- 监控和告警
- 自动化与人工复验

八、无问题维度声明
九、红黄绿验收结论及触发依据
```
